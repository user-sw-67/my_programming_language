# Свой язык программирования

## Расширение для VSCode
Из папки /atm-vscode-extension установить в расширения .vsix файл

ЭТАП 1: Лексический анализ (Lexer)
    Что делает: Режет строку на токены (TokenType).

    Результат: Поток токенов.

ЭТАП 2: Синтаксический анализ (Parser)
    Что делает: Проверяет грамматику (вложенность скобок, порядок операторов).

    Результат: Построение AST (Abstract Syntax Tree), которое ты уже реализовал.

ЭТАП 3: Семантический анализ (Двупроходный)
    Здесь начинается магия "осмысления". Поскольку у тебя динамика, мы не можем проверить всё (например, что в a / b переменная b не окажется строкой в рантайме), но мы можем проверить структуру.

        Шаг 3.1: Подготовка инфраструктуры
        Перед обходом дерева тебе нужны две системы:

        Diagnostic Engine (Сборщик ошибок): Класс, куда ты будешь записывать ошибки. Это позволит не вылетать на первой же опечатке, а собрать их все.

        Symbol Table (Таблица символов): Стек объектов Scope. Каждый Scope хранит информацию о переменных: имя, is_const, is_function, is_defined.

        Шаг 3.2: Первый проход — Name Resolution (Регистрация)
        Мы обходим дерево только по "верхушкам".

        Что ищем: FunctionNodeAST, ClassNodeAST, глобальные MakeNodeAST.

        Что делаем: Записываем их имена в GlobalScope.

        Зачем: Чтобы если в строке №5 ты вызываешь функцию foo(), которая объявлена в строке №100, семантика не выдала ошибку "Функция не найдена".

        Варианты: Если это пропустить, язык станет "однопроходным" (как старый Паскаль или Си), где порядок функций критически важен.

        Шаг 3.3: Второй проход — Глубокий анализ (Validation)
        Теперь основной SemanticVisitor идет по каждому узлу сверху вниз.

            Scope Tracking (Слежка за контекстом):

                Визитор заходит в BlockNodeAST или FunctionNodeAST.

                Создает новый Scope.

                Все новые MakeNodeAST внутри этого блока регистрируются только в этом локальном Scope.

                Почему: Чтобы переменные внутри функции не конфликтовали с глобальными.

            Identifier Resolution (Поиск имен):

                Встречаем IdentifierNodeAST. Ищем его имя сначала в текущем Scope, потом у родителя, и так до Global.

                Что отлавливаем: Ошибку Undeclared Identifier (использование переменной без make).

            Mutability Check (Проверка констант):

                Встречаем BinaryOperationNodeAST с оператором = (присваивание).

                Проверяем левую часть. Если это переменная, ищем её в таблице символов.

                Если у неё флаг is_const == true — кидаем ошибку.

                Отлавливаем: Попытку записи в константу.

            Control Flow Validation (Потоки управления):

                В визиторе заведены счетчики: loop_depth и function_depth.

                В WhileNodeAST/ForNodeAST увеличиваем loop_depth++.

                Встретили BreakNodeAST? Если loop_depth == 0 -> ошибка "Break вне цикла".

                Встретили ReturnNodeAST? Если function_depth == 0 -> ошибка "Return вне функции".

            Type Consistency (Опционально для динамики):

                У тебя в MakeNodeAST есть список type_names.

                Если пользователь написал make x: int = "string", ты сравниваешь тип литерала (который ты уже сохранил в узле) с заявленным типом.

                Отлавливаем: Несоответствие типов при инициализации.

ЭТАП 4: Оптимизация AST (Transformation)
    Когда семантика подтвердила, что код корректен, мы можем сделать дерево "быстрее".

        Constant Folding (Свертка констант):

            Если видим BinaryOperationNodeAST (например, 2 + 3), заменяем этот узел на один LiteralNodeAST(5).

            Зачем: Чтобы рантайм не тратил время на сложение чисел, которые известны заранее.

        Desugaring (Удаление сахара):

            Твой оператор Pipe |> или сложные конструкции превращаются в простые вызовы функций.

            Пример: data |> func() переписывается в func(data).

        Symbol Indexing:

            Вместо того чтобы хранить имя "my_var" в IdentifierNodeAST, мы записываем туда его индекс в таблице символов (например, local_0, local_1).

            Зачем: Интерпретатор будет мгновенно брать значение из массива по индексу, а не искать строку в хэш-таблице.
