ATOM: ТИПИЗАЦИЯ ЯЗЫКА (GRADUAL TYPING)
========================================

ATOM использует гибридную типизацию (gradual typing): часть переменных
типизирована динамически ("auto"), часть — статически (явный `-> Type`).
Обе формы могут свободно соседствовать в одной программе и даже в одном
выражении.


1. ДИНАМИКА (auto)
---------------------

`make x = 0;`               -- тип x неявно "auto"
`make x;`                   -- без типа И без инициализатора -- тоже "auto"
`func f(a) {...}`           -- параметр a без `-> Type` имеет тип "auto"

Если переменная/параметр имеют тип "auto":
  - семантический анализ НЕ проверяет совместимость типов при инициализации,
    вызовах, return и т.п. (см. правило (b) в docs/casting.txt) — проверка
    типов откладывается на этап выполнения программы (когда появится
    интерпретатор);
  - в "auto"-переменную можно положить значение любого типа, в т.ч. в разные
    моменты времени — разные типы (полностью динамическая переменная).


2. СТРОГОСТЬ (явные типы)
----------------------------

`make x -> Int = 0;`        -- x навсегда зафиксирован как Int
`func f(a -> Int) {...}`    -- параметр a навсегда Int

Если тип указан явно:
  - тип фиксируется один раз и не может быть переопределён;
  - AnalysisVisitor обязан выдать ошибку компиляции при попытке записать в
    такую переменную/параметр/поле значение несовместимого типа (правило
    can_cast, см. docs/casting.txt);
  - имя типа должно резолвиться в класс (встроенный или пользовательский),
    иначе — ошибка "Использование необъявленного типа данных".


2а. ОСОБЫЙ СЛУЧАЙ: ЯВНОЕ `-> Null` КАК ТИП-ПРИЁМНИК
--------------------------------------------------------

`Null` — обычный встроенный класс (как Int/Str/Array), но с особым правилом
can_cast (см. docs/casting.txt, пункт "g"): если expected-тип == "Null", каст
разрешён для значения ЛЮБОГО actual-типа. Поэтому переменная, явно
объявленная как `make x -> Null;`, ведёт себя как универсальный "приёмник":

    make x -> Null;        -- тип x навсегда "Null" (явная аннотация)
    x = 10;                 -- ок (Int -> Null разрешён)
    x = "hello";             -- ок (Str -> Null разрешён)
    x = true;                -- ок (Bool -> Null разрешён)

Это НЕ путать с "auto" (раздел 1): `make x;` (без `-> Type`) остаётся
"auto" по умолчанию, не "Null". `Null` как тип получается ТОЛЬКО явной
аннотацией `-> Null`. Обратное направление (Null -> что-то) запрещено для
примитивов: `make s -> Str = null;` — ошибка (см. таблицу в
docs/casting.txt).


3. КАК ВЫВОДИТСЯ ТИП ВЫРАЖЕНИЯ (AnalysisVisitor::infer_type_expression)
--------------------------------------------------------------------------

Тип произвольного выражения вычисляется рекурсивно (с мемоизацией по
указателю AST-узла, т.к. одно и то же выражение может быть переспрошено
несколько раз — например, и при проверке `make`, и при проверке аргумента
вызова). Возвращаемое значение — имя типа как строка ("Int", "Double",
"Str", "Bool", "Null", "Function", "Type", имя класса) либо "auto", если
тип не выводится статически.

Правила вывода по видам узлов AST:

  Литерал (LiteralNodeAST)
      Int/Double/Str/Bool/Null по типу токена литерала. Однозначно.

  Идентификатор (IdentifierNodeAST)
      - ссылается на функцию      -> "Function"
      - ссылается на класс        -> "Type"   (имя самого класса как значение,
                                                 см. ниже про класс Type)
      - ссылается на переменную   -> объявленный тип переменной (или "auto")

  Унарная операция
      `!x`  -> "Bool" всегда (независимо от типа x)
      иначе -> тип операнда (`-x` имеет тот же тип, что и x)

  Бинарная операция (BinaryOperationNodeAST), по группам операторов:
      `.`/`?.`                    -> тип поля/метода-как-значения через
                                      обращение к классу левой части (см. ниже
                                      "Доступ к членам класса")
      `=`,`+=`,`-=`,...            -> тип ЛЕВОЙ части (тип присваивания
                                      равен типу переменной, в которую пишут)
      `==`,`!=`,`<`,`>`,`<=`,`>=`,
      `&`,`|`                      -> "Bool" всегда
      арифметика (`+`,`-`,`*`,`/`,
      `//`,`^`,`%`)                -> "auto", если любая сторона "auto";
                                      "Str", если обе стороны Str и оператор `+`;
                                      "Double", если хоть одна сторона Double;
                                      "Int", если хоть одна сторона Int;
                                      "Int", если обе стороны Bool (true+true=2);
                                      иначе "auto"

  Тернарный оператор `cond ? a : b`
      тип a, если тип a == тип b, иначе "auto" (типы веток разошлись —
      статически вывести общий тип не получается)

  Вызов (CallOperationNodeAST)
      - вызов идентификатора-класса (конструктор) -> имя класса
      - вызов идентификатора-функции              -> объявленный return_type
                                                       функции
      - вызов метода через `.`/`?.`                -> return_type метода
                                                       (НЕ "Function" —
                                                       см. ниже про разницу
                                                       между обращением к
                                                       члену и его вызовом)
      - иначе                                      -> "auto"

  Диапазон (RangeOperationNodeAST, `a..b`/`a..b step c`)
      -> "Range" -- встроенный класс стандартной библиотеки (как и Int,
      Function, Type). Участвует в can_cast-проверках на общих основаниях
      (совпадение с самим собой, null -> Range), а в `for (x in a..b)`
      проверяется итерируемость класса Range точно так же, как для любого
      другого класса -- через метод iter() (см. docs/construct_rules.txt,
      раздел 9). Никакого специального случая "диапазон всегда разрешён в
      for" в коде больше нет: Range разрешён в for ровно потому, что у
      него, как и у Array, есть метод iter().


4. ДОСТУП К ЧЛЕНАМ КЛАССА: РАЗНИЦА МЕЖДУ "ВЗЯТЬ" И "ВЫЗВАТЬ"
------------------------------------------------------------------

`obj.method`     -- это ОБРАЩЕНИЕ к члену класса. Если method — это метод
                    (функция), тип такого выражения — "Function" (мы взяли
                    функцию как значение, ещё не вызвав её).
`obj.method()`   -- это ВЫЗОВ метода. Тип результата — return_type метода,
                    а не "Function" (мы получили РЕЗУЛЬТАТ вызова, а не
                    саму функцию).
`obj.field`      -- тип объявленного поля (для полей разницы между "взять"
                    и "вызвать" нет, т.к. поле не вызывается).

Обе формы (включая вложенные цепочки `obj.method().field`) разрешаются
через поиск члена в дереве наследования класса (по цепочке `parent_name`,
от текущего класса до самого базового), а не только в собственном scope
класса — поэтому унаследованные поля/методы видны точно так же, как
объявленные напрямую.


5. ВСТРОЕННЫЕ "ТИПЫ-КЛАССЫ" Function И Type
-----------------------------------------------

Помимо обычных примитивов (Int/Double/Bool/Str/Null) стандартная библиотека
регистрирует два специальных класса:

  Function
      Тип значения "функция как объект". Любой идентификатор, ссылающийся
      на функцию (встроенную или пользовательскую), имеет тип Function.
      Пример: `make f -> Function = print;` — переменной f можно присвоить
      ТОЛЬКО функцию (любое другое значение, включая числа и строки,
      отклоняется на этапе компиляции).

  Type
      Тип значения "класс как объект" — позволяет хранить, передавать и
      сравнивать сами типы, а не их экземпляры. Любой идентификатор,
      ссылающийся на класс (Int, Str, Array, пользовательский класс),
      имеет тип Type. Пример: `make t -> Type = Int;` — переменной t можно
      присвоить ТОЛЬКО класс (имя типа), но не его экземпляр.
      Используется также как тип первого параметра runtime-функции
      primitive_cast и как тип результата typeof() — см. docs/casting.txt.

И Function, и Type — обычные классы с точки зрения правил совместимости
типов (can_cast): кастов "из" и "в" них нет вообще, кроме совпадения
с самим собой и null (см. docs/casting.txt, пункт "д/е").


6. ВНУТРЕННЕЕ ПРЕДСТАВЛЕНИЕ (Value, только для будущего интерпретатора)
----------------------------------------------------------------------------

Класс Value (include/semantics/value.hpp) хранит исполняемое значение как
std::variant. Function и Type представлены одной и той же по форме
обёрткой `ValueTypeList::SymbolRef<Tag>` (поле `SymbolInfo* info`),
параметризованной тегом (`FunctionTag`/`TypeTag`) — это сделано специально,
чтобы variant мог различать "значение-функция" от "значение-тип", даже
если внутри лежит указатель на один и тот же SymbolInfo. Семантический
анализ (AnalysisVisitor) с Value не работает вообще — он оперирует только
именами типов (строками) и SymbolInfo; Value предназначен исключительно
для будущего бэкенда-интерпретатора.
