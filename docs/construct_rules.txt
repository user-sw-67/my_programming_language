ATOM: СЕМАНТИЧЕСКИЕ ПРАВИЛА КОНСТРУКЦИЙ
===========================================

В отличие от docs/constructs.txt (что МОЖНО написать синтаксически), этот
файл описывает, что из написанного семантический анализатор (3 прохода:
DefinitionVisitor -> AnalysisVisitor -> OptimizationVisitor, см. CLAUDE.md)
считает корректным, а что -- ошибкой или предупреждением. Типовые проверки
(совместимость типов) описаны отдельно в docs/casting.txt и docs/typing.txt
-- здесь только структурные/контрольно-потоковые правила.


1. ОБЛАСТИ ВИДИМОСТИ И ИМЕНА
--------------------------------

  - Каждый `{ }` (включая тела if/while/for/try/match/test) -- отдельная
    область видимости (Scope) с обратной ссылкой на родителя.
  - Повторное `make` с тем же именем в ТОЙ ЖЕ области -- ошибка
    "Идентификатор уже определен в этой области видимости".
  - Повторное `make` с тем же именем во ВЛОЖЕННОЙ области -- разрешено,
    это затенение (shadowing), а не конфликт.
  - Использование идентификатора, который не резолвится ни в текущей, ни в
    родительских областях (вплоть до глобальной и стандартной библиотеки)
    -- ошибка "Использование необъявленного идентификатора".
  - Переопределять идентификаторы стандартной библиотеки (is_std) запрещено
    в любой области.


2. КОНСТАНТЫ (const)
-------------------------

  - `make const x = ...;` фиксирует x как неизменяемую.
  - Присваивание (`=` и составные `+=`, `-=`, ...) в простую const-переменную
    -- ошибка "Нельзя переприсвоить значение константе".
  - Поля класса, объявленные как `const`, защищены так же: присваивание
    `obj.field = ...` (в т.ч. через цепочки `obj.method().field = ...`,
    т.к. владелец поля вычисляется через полноценный вывод типа выражения,
    а не только для простых идентификаторов) -- ошибка "Нельзя переприсвоить
    значение константному полю".
  - `this` и `super` внутри класса -- неявные const-переменные (заданы
    автоматически), их также нельзя переприсваивать.


3. ЦЕЛЕВОЕ ВЫРАЖЕНИЕ ПРИСВАИВАНИЯ
--------------------------------------

  - Левой частью `=`/составного присваивания может быть ТОЛЬКО:
      - простой идентификатор (IdentifierNodeAST), либо
      - доступ к члену через `.`/`?.` (BinaryOperationNodeAST с этим
        оператором).
  - Любое другое выражение слева (например, литерал, вызов функции,
    результат арифметики) -- ошибка "Левая часть выражения не может быть
    целью для присваивания".


4. УПРАВЛЕНИЕ ПОТОКОМ: return / break / continue / throw / assert
------------------------------------------------------------------------

  - `return` допустим только внутри тела функции (включая методы класса);
    иначе -- ошибка "Конструкция return невозможна вне функции".
  - Тип возвращаемого значения (или Null для `return;` без значения)
    обязан быть совместим (can_cast) с объявленным return_type функции.
  - `break`/`continue` допустимы только внутри цикла (while/do-while/for),
    отслеживается флагом in_cycle, который сбрасывается на выходе из цикла;
    иначе -- "Конструкция break/continue невозможна вне цикла".
  - `throw` допустим только внутри функции; иначе -- "Конструкция throw
    невозможна вне функции".
  - `assert` допустим только внутри тела `test`; иначе -- "Конструкция
    assert невозможна вне тестов".


4а. ВСЕ ПУТИ ФУНКЦИИ ОБЯЗАНЫ ВОЗВРАЩАТЬ ЗНАЧЕНИЕ (always_returns)
------------------------------------------------------------------------

  Если у функции (или метода класса) явно указан return_type, отличный от
  "auto" и "Null", AnalysisVisitor проверяет, что АБСОЛЮТНО ВСЕ пути
  исполнения тела гарантированно проходят через `return` (или `throw` --
  он тоже "выходит" из функции, просто через исключение, а не значение).
  Если есть хотя бы один путь, на котором функция может "просто кончиться"
  без return -- ошибка "Не все пути функции ... завершаются оператором
  return: возможен выход без значения типа ...".

  Правила вывода "гарантированно завершается" (always_returns) по видам
  statement:
    - `return ...;` / `throw ...;`           -- да, всегда
    - блок `{ s1; s2; ...; sn; }`             -- да, если ХОТЯ БЫ ОДИН из
                                                  statement'ов в нём
                                                  гарантированно завершается
                                                  (всё, что после него --
                                                  уже вырезано как
                                                  недостижимый код, см. п.5)
    - `if (c) A else B`                       -- да, только если ЕСТЬ ветка
                                                  `else` И обе ветки (A и B)
                                                  гарантированно завершаются.
                                                  `if` без `else` -- никогда
                                                  не гарантирует (есть путь
                                                  "условие не выполнилось")
    - `while`/`do-while`/`for`                -- никогда не гарантирует
                                                  (компилятор не пытается
                                                  доказывать, что цикл
                                                  исполнится хотя бы раз)
    - `try { } catch { } finally { }`         -- да, если `finally`
                                                  гарантированно завершается
                                                  САМ ПО СЕБЕ (он выполнится
                                                  в любом случае), ИЛИ если
                                                  `try`-блок гарантированно
                                                  завершается И (нет `catch`
                                                  ИЛИ `catch`-блок тоже
                                                  гарантированно завершается)
    - `match { case ...: A; ... default: Z }` -- да, только если ЕСТЬ
                                                  `default` И КАЖДЫЙ
                                                  case/default гарантированно
                                                  завершается
    - всё остальное (отдельный вызов, `make`,
      `break`/`continue` и т.п.)              -- нет

  `break`/`continue` НЕ считаются "гарантированным завершением" в этой
  проверке (в отличие от п.5): они завершают цикл, а не функцию, и
  исполнение продолжается дальше в окружающем блоке.


5. НЕДОСТИЖИМЫЙ КОД
------------------------

  - Если внутри блока `{ }` встречается `return`/`break`/`continue`/`throw`,
    а ПОСЛЕ него в этом же блоке есть ещё операторы -- они физически
    вырезаются из AST (BlockNodeAST::statements обрезается до точки
    завершения), а пользователю выдаётся ПРЕДУПРЕЖДЕНИЕ (не ошибка):
    "Код после return/break/continue/throw недостижим, поэтому вырезан
    из дерева".
  - Эта проверка работает только на верхнем уровне одного блока: она НЕ
    анализирует ветвления (если из if-ветки нет выхода, код после всего
    if/else не считается недостижимым только по этой причине) -- этим
    занимается отдельная, более глубокая проверка always_returns (см. п.4а).


6. УСЛОВИЯ ДОЛЖНЫ БЫТЬ Bool
--------------------------------

  Проверяется во всех местах, где конструкция языка ожидает условие:
    - `if (...)` / `elif (...)`
    - `while (...)` / `do { } while (...)`
    - тернарный оператор `cond ? a : b`
    - `when (...)` -- guard-условие функции
  Если выведенный тип условия не "Bool" и не "auto" -- ошибка "Условие
  должно иметь тип Bool". Для "auto"-выражений проверка не выполняется
  (динамика, см. docs/typing.txt) -- ошибка может проявиться только в
  рантайме.


7. ВЫЗОВЫ: КОЛИЧЕСТВО И ТИПЫ АРГУМЕНТОВ
--------------------------------------------

  Для каждого вызова (обычной функции, метода через `.`/`?.`, конструктора
  класса через `ИмяКласса(...)`):

    а) Количество аргументов. Пусть count_args -- объявленное число
       параметров, count_elem_default -- сколько из них имеют значение по
       умолчанию, is_ellipsis_args -- есть ли вариативный `...`-параметр.
       Минимально необходимое число аргументов = count_args -
       count_elem_default (минус 1, если последний параметр вариативный --
       вариативный параметр сам по себе не обязателен).
         - без `...`: фактическое число аргументов должно быть в диапазоне
           [минимум, count_args];
         - с `...`: фактическое число аргументов должно быть >= минимума
           (сверху не ограничено).
       Несоответствие -- ошибка "Количество передаваемых аргументов (N) ...
       не соответствует ... сигнатуре".

    б) Типы аргументов. Если у параметра функции/метода ЯВНО указан тип
       (`a -> Int`), то тип переданного аргумента (выведенный через
       infer_type_expression) обязан быть can_cast-совместим с типом
       параметра; иначе -- ошибка "Аргумент N имеет тип ..., ожидался
       тип ...". Для параметров без явного типа (auto) проверка типов
       не выполняется.

  Проверка типов аргументов выполняется ТОЛЬКО для функций/методов,
  написанных на самом ATOM (у которых есть AST тела, body_ast != nullptr).
  Встроенные функции стандартной библиотеки (is_built_in) проверяются
  только по количеству аргументов -- у них нет AST-параметров с типами.

  Если вызывается идентификатор, который не является функцией или классом
  (например, переменная) -- ошибка "Идентификатор ... не может быть
  вызван".


8. ЗНАЧЕНИЯ ПО УМОЛЧАНИЮ У ПАРАМЕТРОВ
-------------------------------------------

  Если параметр одновременно имеет явный тип и значение по умолчанию
  (`a -> Int = 5`), тип значения по умолчанию обязан быть can_cast-
  совместим с объявленным типом параметра -- иначе ошибка "Значение по
  умолчанию имеет тип ..., ожидался тип ...".


9. ИТЕРИРУЕМОСТЬ for (... in ...)
---------------------------------------

  Единственный признак, по которому ЛЮБОЙ класс (встроенный или
  пользовательский) считается итерируемым -- наличие у него метода `iter`.
  Никаких особых случаев "для диапазона" или "для массива" в семантике нет:
  диапазон `a..b` типизируется как обычный класс "Range" (см.
  docs/typing.txt), который, как и встроенный класс "Array", сам объявляет
  метод `iter()` в стандартной библиотеке -- поэтому `for (x in a..b)` и
  `for (x in array)` проверяются ОДНИМ И ТЕМ ЖЕ кодом, без dynamic_cast на
  конкретный тип узла AST.

  Алгоритм проверки `for (x in выражение)`:
    1. Вычислить тип выражения через infer_type_expression. Если тип
       "auto" -- проверка не выполняется (динамика, ошибка возможна только
       в рантайме).
    2. Если тип резолвится в класс (SymbolInfo::type == CLASS) и у класса
       взведён кэш-флаг SymbolInfo::is_iter_obj -- всё хорошо.
    3. Иначе -- ошибка "Тип ... не является итерируемым: класс должен
       иметь метод iter()".

  SymbolInfo::is_iter_obj -- ЧИСТО оптимизация скорости доступа (булево
  значение вместо поиска метода `iter` по scope на каждое использование в
  `for`), а не отдельный механизм регистрации. Он вычисляется один раз:
    - для встроенных классов (Range, Array) -- сразу при регистрации в
      BuildInManager (там же, где объявлен метод iter()), см.
      src/addition/builds_manager.cpp;
    - для пользовательских классов -- в DefinitionVisitor::visit(ClassNodeAST&)
      сразу после построения scope класса: флаг истинен, если в собственном
      scope класса есть символ "iter", ЛИБО если базовый класс (`extends`)
      уже итерируем (is_iter_obj наследуется по цепочке так же, как и сам
      метод iter() через обычный поиск членов).

  Чтобы сделать СВОЙ класс итерируемым, достаточно объявить в нём метод
  `iter() { ... }` -- никакой дополнительной регистрации не требуется.


10. PATTERN MATCHING (match/case)
----------------------------------------

  - Допустим максимум один `default` -- повторный `default` -- ошибка
    "Больше одной инструкции default невозможно".
  - Тип значения каждого `case` должен быть can_cast-совместим (в любую
    сторону: ожидаемый-к-фактическому ИЛИ фактический-к-ожидаемому) с
    типом сопоставляемого выражения `match (значение)`; иначе -- ошибка
    "Тип значения case (...) несовместим с типом сопоставляемого
    выражения (...)".
  - Если значение `case` -- литерал, его пара (тип литерала + текстовое
    значение) сравнивается с уже встреченными в этом `match`; повтор --
    ошибка "Повторяющееся значение в case: ...". Для не-литеральных
    значений `case` (выражений, диапазонов) дублирование не проверяется.


11. КЛАССЫ
---------------

  - Конструктор `new()` и деструктор `delete()` -- зарезервированные
    имена. Если пользователь не объявил их явно, DefinitionVisitor создаёт
    пустые версии автоматически (0 аргументов, public, не static). Если
    пользователь объявил `new`/`delete` как ПОЛЕ (не метод) -- ошибка
    "Объявлять поля с именем new или delete запрещено".
  - У деструктора `delete()` не может быть параметров -- иначе ошибка
    "Деструктор класса ... не должен иметь параметров".
  - `new`/`delete` не могут иметь модификатор static -- если он есть,
    выдаётся ПРЕДУПРЕЖДЕНИЕ, и модификатор сбрасывается.
  - `new`/`delete` обязаны быть public -- иначе ПРЕДУПРЕЖДЕНИЕ, модификатор
    принудительно меняется на public.
  - Обращение к нестатическому полю/методу из статического контекста
    (внутри static-метода) -- ошибка "Нельзя использовать нестатическое
    поле или метод ... в статическом контексте".
  - `extends`: класс не может наследоваться от самого себя -- ошибка
    "Класс ... не может быть наследником самого себя". Базовый класс
    должен существовать и быть классом -- иначе ошибка "Базовый класс ...
    не найден".
  - Поиск членов класса (полей/методов) для `.`/`?.`-доступа, для проверки
    типов аргументов вызова и для проверки const учитывает цепочку
    наследования: если член не найден в собственном scope класса, поиск
    продолжается у родителя (через parent_name), и так далее до самого
    базового класса.
  - `super` -- внутри класса, объявленного с `extends`, регистрируется как
    функция (`SymbolType::FUNCTION`, не переменная), привязанная к сигнатуре
    конструктора `new()` родительского класса (количество аргументов,
    значения по умолчанию, `...`-параметр и типы параметров копируются у
    родительского `new` на этапе DefinitionVisitor). Поэтому `super(...)`
    внутри `new()` потомка проверяется AnalysisVisitor ТОЧНО так же, как
    обычный вызов функции (`SEM_2504`/`SEM_2106` при несовпадении сигнатуры).
    `super`, как и `this`, помечен `is_const` -- `super = ...;` запрещено
    (`SEM_2302`). Реальный вызов конструктора родителя (тело) исполняется
    будущим интерпретатором -- семантика на сегодня проверяет только
    сигнатуру вызова.
  - Контроль доступа (`private`/`protected`/`public`) применяется при ЛЮБОМ
    обращении к члену класса через `.`/`?.` (чтение значения, вызов метода,
    левая часть присваивания) -- единая точка проверки:
    `AnalysisVisitor::resolve_member_symbol` -> `check_member_access`.
      - `public` -- всегда разрешено.
      - `private` -- разрешено ТОЛЬКО если код, выполняющий обращение,
        находится внутри ТОГО ЖЕ класса, который объявил член (`SymbolTable::
        get_current_class() == владелец`). Не наследуется: метод класса-
        потомка не может прочитать `private`-поле родителя. Нарушение --
        ошибка "Доступ к private-члену ... запрещён снаружи класса"
        (`SEM_2205`).
      - `protected` -- разрешено внутри класса-владельца И внутри любого
        класса-потомка (по цепочке `parent_name`). Нарушение -- ошибка
        "Доступ к protected-члену ... запрещён снаружи класса и его
        наследников" (`SEM_2206`).
    Проверка применяется только когда тип выражения слева от `.`/`?.`
    выведен статически (не "auto") -- для auto-выражений, как и везде в
    gradual typing, проверка типов/доступа откладывается до рантайма.


12. ИМПОРТЫ (use)
----------------------

  - Циклический импорт файлов (A импортирует B, B импортирует A) --
    ошибка "Обнаружен циклический импорт".
  - Выборочный импорт конкретных имён (`use { a, b } from "..."`)
    запрещает транзитивный импорт: если запрашиваемое имя само было
    импортировано в файле-источнике (а не объявлено там), будет выдано
    ПРЕДУПРЕЖДЕНИЕ с просьбой импортировать имя из исходного файла, а не
    через посредника.
  - Имя, отсутствующее в указанном модуле/файле -- ошибка "Идентификатор
    ... не найден в модуле ...".


13. ИДЕИ НА БУДУЩЕЕ (НЕ РЕАЛИЗОВАНО)
------------------------------------------

Список пунктов, которые обсуждались, но пока не реализованы в семантике
(зафиксировано здесь, чтобы не терять контекст обсуждения):

  - Проверка типа элементов итерации в for (что именно отдаёт `iter()`) --
    сейчас проверяется только сам факт итерируемости класса (наличие
    метода iter(), см. раздел 9), но не тип элементов, которые будут
    получены в `for (x in obj)`. Для этого потребуется договориться о
    протоколе итерации (что возвращает iter(): сам объект, отдельный
    итератор с next()/has_next(), и т.п.) -- естественно сделать это
    вместе с реализацией интерпретатора, а не раньше.
  - Явный оператор каста на уровне синтаксиса (`x as Int`), который бы
    транслировался в вызов primitive_cast (см. docs/casting.txt, раздел 3).
  - Проверка сигнатур (количества параметров) для самого метода `iter()`
    -- сейчас семантика не проверяет, что iter() объявлен без обязательных
    параметров; некорректная сигнатура будет замечена только в момент
    попытки его вызвать рантаймом (которого пока нет).
  - Реальное исполнение `super(...)` (вызов тела конструктора родителя) --
    на сегодня семантика проверяет только сигнатуру вызова (раздел 11),
    само исполнение -- задача будущего интерпретатора.
