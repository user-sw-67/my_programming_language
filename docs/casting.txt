ATOM: ПРАВИЛА КАСТОВ (ПРИВЕДЕНИЯ ТИПОВ)
=========================================

В языке ATOM есть ДВА независимых механизма приведения типов, которые не
нужно путать между собой:

  1. Неявная совместимость типов (compile-time, can_cast) — проверяется
     семантическим анализатором на этапе компиляции, нигде не "конвертирует"
     значение, а только решает, можно ли значение типа B записать в место,
     ожидающее тип A.
  2. Явное преобразование значений (runtime, primitive_cast/typeof) —
     встроенные функции стандартной библиотеки, которые реально меняют
     представление значения (например, парсят строку "42" в число 42).

Эти два набора правил СОЗНАТЕЛЬНО различаются: правило компиляции отвечает
на вопрос "можно ли положить B туда, где объявлено A без явного
преобразования", а runtime-функция primitive_cast отвечает на вопрос
"могу ли я явно попросить преобразовать одно значение в другое". Например,
Str -> Int запрещён как неявный каст (нельзя написать `make x -> Int = "5";`),
но разрешён как явное преобразование (`primitive_cast(Int, "5")`).


1. НЕЯВНАЯ СОВМЕСТИМОСТЬ ТИПОВ (AnalysisVisitor::can_cast)
------------------------------------------------------------

Используется во всех местах, где компилятор сверяет "ожидаемый тип" против
"фактического типа выражения":
  - `make x -> Type = expr;`               (инициализация переменной/поля)
  - значения по умолчанию параметров       (`func f(a -> Int = 5)`)
  - аргументы вызовов функций/методов/конструкторов с явно типизированными
    параметрами
  - оператор `return` внутри функции с явно указанным `-> Type`
  - значения `case` в `match` относительно типа сопоставляемого выражения

Сигнатура: `can_cast(expected, actual) -> bool`, где `expected` — это
объявленный (ожидаемый) тип, `actual` — выведенный тип фактического
значения (см. docs/typing.txt про то, как типы выводятся).

Правила (в порядке проверки):

  a) expected == actual                    -> разрешено всегда (тривиально)
  b) expected == "auto" ИЛИ actual == "auto" -> разрешено всегда
     (гибридная типизация: динамика "обтекает" любую строгую сторону,
     проверка типа в этом случае откладывается до рантайма интерпретатора)
  c) числовые/булевы примитивы: разрешены ОБЕ стороны для каждой из пар
        Int   <-> Double
        Bool  <-> Double
        Bool  <-> Int
     (аналог неявных числовых преобразований С++ между арифметическими
     типами; буквально совпадающие типы уже отброшены пунктом "a")
  d) actual == "Null" -> разрешено, ЕСЛИ expected — это класс (встроенный
     или пользовательский, включая Function/Type/Array/...). Это аналог
     неявного приведения nullptr к любому указательному/ссылочному типу
     в С++: null можно положить в переменную любого классового типа, но
     NELZЯ положить в Int/Double/Bool/Str (для них own null-состояния нет).
  e) во всех остальных случаях, КОГДА actual — класс: каста нет вообще.
     В частности:
       - НЕТ восходящего каста "потомок -> предок" (если A -- предок B,
         в переменную типа A всё равно нельзя положить значение типа B без
         явного преобразования). Присваивать можно только значение РОВНО
         того же класса, что и объявленный тип.
       - Function и Type — обычные классы с точки зрения can_cast: не
         кастятся ни к чему, кроме самих себя (и принимают null).
  f) Str кастится только сам в себя (никаких неявных Str<->Int/Double/Bool).
  g) expected == "Null" -> разрешено ВСЕГДА, для значения любого actual-типа
     (проверяется раньше пункта "d", т.к. не зависит от того, класс ли
     actual). `Null` — единственный тип-"приёмник" в языке: переменную,
     явно объявленную как `make x -> Null;`, можно после этого свободно
     заполнять значением любого типа (`x = 5; x = "hello"; x = true;` —
     всё валидно). Это НЕ влияет на то, чем СЧИТАЕТСЯ переменная без
     явного типа (`make x;` остаётся "auto", см. docs/typing.txt) — `Null`
     как тип получается ТОЛЬКО явной аннотацией `-> Null`.

Таблица итогов для примитивов (строка = expected, столбец = actual; "+" —
разрешено неявно, "-" — запрещено; "auto" в любой позиции всегда "+"):

            Int   Double  Bool   Str   Null
   Int       +      +      +      -     -
   Double    +      +      +      -     -
   Bool      +      +      +      -     -
   Str       -      -      -      +     -
   Null      +      +      +      +     +   (expected == Null -> приёмник для всего)
   Null      -      -      -      -     +

Для классов (включая Function, Type, Array и пользовательские классы):
   - X (expected) <- X (actual)      : +
   - X (expected) <- Null            : +
   - X (expected) <- Y, X != Y       : -   (даже если Y — наследник X)


2. ЯВНОЕ ПРЕОБРАЗОВАНИЕ ЗНАЧЕНИЙ (std: typeof / primitive_cast)
------------------------------------------------------------------

Это runtime-функции из стандартной библиотеки (`@std`, доступны без `use`),
предназначенные для интерпретатора. Они работают со значениями `Value`,
а не с объявленными типами AST, и используют намеренно более широкие
правила преобразования, чем can_cast — потому что это явная операция,
о последствиях которой пользователь заявляет осознанно.

  typeof(object) -> Type
      Возвращает значение типа Type, "ссылающееся" на класс значения
      object (Int/Double/Bool/Str/Null/Function/Type или класс объекта).
      В отличие от type(object) (возвращает Str с текстовым описанием),
      typeof возвращает полноценное значение типа Type, которое можно
      положить в переменную (`make t -> Type = typeof(x);`), сравнить
      оператором == с другим Type (`if (typeof(x) == Int) {...}`) или
      передать как первый аргумент в primitive_cast.

  primitive_cast(type, object) -> auto
      Явно преобразует ПРИМИТИВНОЕ значение object (Int/Double/Bool/Str/
      Null) к типу type (значение типа Type, например `Int`, `Str`,
      результат typeof(...), и т.п.). Поддерживаются только примитивные
      целевые типы — Int, Double, Bool, Str, Null; для классов/Function/
      Type выбрасывается TypeError.

      Таблица преобразований (источник -> назначение):

        источник \ назнач.   Int        Double     Bool         Str           Null
        Bool                 0/1        0.0/1.0    как есть     "true"/"false" null
        Int                  как есть   double(x)  x != 0       to_string(x)   null
        Double               int(x)     как есть   x != 0.0     to_string(x)   null
        Str                  std::stoll std::stod  !empty()     как есть       null
                             (или ошибка, если строка не число)
        Null                 0          0.0        false        "null"        null

      Преобразование Str -> Int/Double выбрасывает TypeError, если строка
      не парсится как число. primitive_cast(Null, любое) всегда возвращает
      Value() (null) — это единственное преобразование, доступное из
      абсолютно любого примитива.

  is_primitive(object) -> Bool
      true, если object — Int/Double/Bool/Str/Null (то есть один из типов,
      которые умеет обрабатывать primitive_cast).


3. ПОЧЕМУ ДВА НАБОРА ПРАВИЛ, А НЕ ОДИН
-----------------------------------------

Компилятору важно ловить ошибки до запуска программы: если в строго
типизированную переменную пытаются положить значение несовместимого типа,
это почти всегда ошибка программиста, а не намеренное действие — поэтому
can_cast строгий и не пытается "угадать", как привести Str к Int.

primitive_cast, наоборот, вызывается явно и предназначен именно для
конвертации представления (разбор строки, форматирование числа и т.п.),
поэтому он позволяет себе намеренно более широкий набор преобразований.
Если в будущем потребуется языковая конструкция явного каста (например,
`x as Int`), она должна транслироваться именно в вызов primitive_cast,
а не пытаться обойти правила can_cast.
